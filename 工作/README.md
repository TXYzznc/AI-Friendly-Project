## 🔄 完整工作流

### Phase 1️⃣: 策划讨论

**位置**：`1.策划/NN.功能名/`

**输入**：你的口头需求
**输出**：`策划案.md` + 可选的 `讨论记录.md`

**流程**：
```
你描述需求
  ↓
我们进行多轮讨论（clarify 细节）
  ↓
你确认满意
  ↓
保存 策划案.md 到 1.策划/NN.功能名/
  ↓
→ 进入 Phase 2️⃣
```

**策划案格式**（Markdown，思维导图式）：
```markdown
# NN.功能名

## 基本信息
- 功能描述：...
- 核心目标：...

## 功能清单
1. 功能A
   1.1 子功能A1
   1.2 子功能A2
2. 功能B
   2.1 子功能B1
   ...

## 约束 & 设计决策
- 性能要求：...
- UI风格参考：...
- ...
```

---

### Phase 2️⃣: 需求拆解 + 美术分析（多Agent协作）

**位置**：`2.需求列表/NN.功能名/` 和 `1.美术/NN.功能名/` 和 `3.正在处理的任务/NN.功能名/`

**输入**：策划案.md
**输出**：
- 需求.md（开发需求）
- 美术需求分析.md + 提示词.md（美术资源）
- OpenSpec 工作目录

**协作模式**：**并行** （需求Agent + 美术Agent 同时工作）

**流程**：

```
策划案.md
  ↓ (多Agent并行)
  ├─ 需求Agent
  │  ├─ 分析策划案
  │  ├─ 拆解成技术需求
  │  ├─ 创建 OpenSpec 变更
  │  ├─ 生成 spec.md + design.md + tasks.md
  │  └─ 保存到 3.正在处理的任务/
  │
  └─ 美术Agent  
     ├─ 分析策划案中的美术需求
     ├─ 生成美术资源清单
     ├─ 生成提示词（基于美术风格规范）
     └─ 保存到 1.美术/NN.功能名/
  ↓
全部完成 → 进入 Phase 3️⃣
```

**生成的文件说明**：

1. **2.需求列表/NN.功能名/需求.md**
   - 开发角度的需求描述
   - 功能点清单
   - 接口设计
   - 配置需求

2. **1.美术/NN.功能名/**
   - 美术需求分析.md - 分析所需的美术资源
   - 提示词.md - AI 生成的提示词（每个资源一个）

3. **3.正在处理的任务/NN.功能名/**
   - .openspec.yaml - OpenSpec 配置
   - proposal.md - 提案（来自需求.md）
   - spec.md - 规范
   - design.md - 设计方案
   - tasks.md - 任务清单 + 进度表
   - 美术资源.md - 美术资源清单（关联 1.美术）

---

### Phase 3️⃣: 追踪执行

**位置**：`3.正在处理的任务/NN.功能名/`

**维护文件**：
- `tasks.md` - 任务清单 + 进度表
- `实现记录/PhaseN.md` - 阶段性实现记录

**流程**：
```
开始实现
  ↓
更新 tasks.md 的进度
  ↓
每个阶段完成后，在 实现记录/ 中记录
  ↓
全部实现完成
  ↓
→ 进入 Phase 4️⃣
```

**tasks.md 格式**：
```markdown
# 任务清单 - NN.功能名

## 总体进度
- [x] 任务1 (完成)
- [ ] 任务2 (进行中)
- [ ] 任务3 (待做)

## 任务详情

### 任务1: ...
- 状态：✅ 完成
- 完成日期：2026-06-12
- 代码位置：Assets/Scripts/...

### 任务2: ...
- 状态：🔄 进行中
- 开始日期：2026-06-12
- 估计完成：2026-06-13
```

### Phase 4️⃣: 自动化测试

**位置**：`3.测试/NN.功能名/`

**输出**：
- 测试方案.md
- 自动化测试脚本.cs（如需要）
- 测试结果.md
- bug报告.md（如有问题）

**流程**：
```
实现完成
  ↓
设计测试方案
  └─ 分析功能复杂度
  └─ 决定是否需要自动化测试
  └─ 如需要，编写测试脚本
  └─ 保存 测试方案.md
  
  ↓
执行测试
  ├─ 运行自动化脚本（如有）
  └─ 记录结果 → 测试结果.md
  
  ↓
有问题？
  ├─ 是 → 记录 bug报告.md → 回到 Phase 3️⃣ 修复
  └─ 否 → 进入 Phase 5️⃣
```

**测试方案.md 格式**：
```markdown
# 测试方案 - NN.功能名

## 测试策略
- 使用自动化测试：是/否
- 原因：...

## 测试场景
1. 场景A：...
   - 预期结果：...
   - 自动化脚本：✅/❌

2. 场景B：...
   ...

## 自动化脚本
- 脚本文件：自动化测试脚本.cs
- 覆盖率：...
```

### Phase 5️⃣: 验证 + 归档

**位置**：`4.已归档任务/NN.功能名_vX/`

**流程**：
```
测试通过，所有问题已修复
  ↓
创建最终验收清单
  ↓
根据版本号，在 4.已归档任务/ 中创建对应文件夹
  ├─ 如果是首次：NN.功能名_v1/
  ├─ 如果是第2次迭代：NN.功能名_v2/
  └─ ...
  ↓
移入完成清单.md
  ↓
归档完成！
```

**完成清单.md 格式**：
```markdown
# 完成清单 - NN.功能名 v1

## 验收状态

  - 测试覆盖：✅

  ...

2026-06-13

## 下一版本计划
（如有计划迭代）
```

## 🔗 与 OpenSpec 的集成

**OpenSpec 的作用**：管理 Phase 2️⃣ 和 Phase 3️⃣ 的任务拆解和追踪

**工作流对应**：
```
Phase 2️⃣ 拆解
  → openspec new change "NN-功能名"
  → 自动生成 proposal.md / spec.md / design.md / tasks.md

Phase 3️⃣ 执行
  → openspec apply-change "NN-功能名"
  → 实现代码 + 更新 tasks.md 进度

Phase 4️⃣ 测试
  → openspec verify-change "NN-功能名"
  → 验证实现完整性

Phase 5️⃣ 归档
  → openspec archive-change "NN-功能名"
  → 完成变更归档
```

---

## 🔀 多Agent协作

在 Phase 2️⃣ 时，同时启动两个 Agent：

1. **需求Agent** ← 拆解功能需求（调用 OpenSpec）
2. **美术Agent** ← 分析美术资源需求（需要美术风格规范）

两个 Agent **并行工作**，完成后汇总。

---

## 📌 版本号管理

文件夹命名规则：
- **首次开发**：`NN.功能名/` → 完成后归档到 `NN.功能名_v1/`
- **第2次迭代**：新建 `3.正在处理的任务/NN.功能名_v2/` → 完成后归档到 `4.已归档任务/NN.功能名_v2/`
- **第3次迭代**：以此类推 `NN.功能名_v3/`

版本递增代表功能的迭代次数。
