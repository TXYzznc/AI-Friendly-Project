# 🚀 快速开始指南

> 新功能开发完整工作流（5 个阶段）

---

## 📋 工作流总览

```
Phase 1️⃣ 策划讨论  →  Phase 2️⃣ 需求拆解  →  Phase 3️⃣ 执行追踪  →  Phase 4️⃣ 测试验证  →  Phase 5️⃣ 归档完成
```

---

## Phase 1️⃣: 策划讨论 (30 分钟)

### 你需要做的
1. **描述需求**（口头）
   ```
   例：我想要一个背包系统，支持 100 格容量，物品可以叠放...
   ```

2. **我会问问题澄清细节**
   ```
   - 背包是否可以升级？
   - 物品叠放上限是多少？
   - UI 风格参考？
   - ...
   ```

3. **多轮讨论，直到你确认**
   ```
   你：这个需求我满意了，可以进行下一步了
   ```

### 输出
- 📄 `工作/1.策划/NN.功能名/策划案.md`

---

## Phase 2️⃣: 需求拆解 (1 小时)

### 系统自动执行的（多Agent协作）

**Agent 1 - 需求Agent**：
```
策划案.md 
  → 分析功能需求
  → 拆解成技术需求
  → 调用 OpenSpec 创建变更
  → 生成 spec.md, design.md, tasks.md
```

**Agent 2 - 美术Agent**：
```
策划案.md
  → 分析美术资源需求
  → 生成美术资源清单
  → 根据美术风格规范生成提示词
  → 保存到 1.美术/ 文件夹
```

### 输出
- 📄 `工作/2.需求列表/NN.功能名/需求.md`
- 📄 `工作/1.美术/NN.功能名/美术需求分析.md`
- 📄 `工作/1.美术/NN.功能名/提示词.md`
- 📂 `工作/3.正在处理的任务/NN.功能名/` (OpenSpec 工作目录)

---

## Phase 3️⃣: 执行追踪 (2-3 小时)

### 你或开发者需要做的

1. **创建代码**
   ```
   根据 design.md 和 tasks.md 实现功能
   ```

2. **我在 tasks.md 中追踪进度**
   ```
   - [x] 任务1 (完成)
   - [x] 任务2 (完成)
   - [ ] 任务3 (进行中)
   - [ ] 任务4 (待做)
   ```

3. **完成时在 实现记录/ 中记录**
   ```
   实现记录/Phase1.md  - 阶段1 完成
   实现记录/Phase2.md  - 阶段2 完成
   ```

### 文件位置
- `工作/3.正在处理的任务/NN.功能名/tasks.md` ← 进度表
- `工作/3.正在处理的任务/NN.功能名/实现记录/` ← 阶段记录

---

## Phase 4️⃣: 自动化测试 (1 小时)

### 系统自动执行

1. **设计测试方案**
   ```
   分析功能复杂度
   → 决定测试方式（自动化 or 手动）
   → 生成测试方案文档
   ```

2. **编写测试脚本**（如需要）
   ```
   InventoryAddItemTests.cs
   InventoryRemoveItemTests.cs
   ...
   ```

3. **运行测试 & 验证**
   ```
   Unity Test Runner → Run All
   → 结果：24/24 PASSED ✅
   ```

### 输出
- 📄 `工作/3.测试/NN.功能名/测试方案.md`
- 📄 `工作/3.测试/NN.功能名/自动化测试脚本.cs`
- 📄 `工作/3.测试/NN.功能名/测试结果.md`

### 有问题？
```
如果测试失败
  → 记录到 bug报告.md
  → 回到 Phase 3️⃣ 修复代码
  → 重新测试
```

---

## Phase 5️⃣: 归档完成 (10 分钟)

### 最终验收

```
所有测试通过 ✅
代码审查通过 ✅
没有遗留问题 ✅
  ↓
生成完成清单
  ↓
移动到 4.已归档任务/
  ↓
✅ 任务完成！
```

### 输出
- 📂 `工作/4.已归档任务/NN.功能名_v1/`
- 📄 `工作/4.已归档任务/NN.功能名_v1/完成清单.md`

---

## 📂 文件夹命名规则

**所有功能文件夹都必须带序号前缀**

```
NN.功能名

NN = 功能的序号（两位数）
例如：
  01.背包系统
  02.战斗系统
  03.任务系统
  ...
```

**版本化管理**

```
首次开发：01.背包系统
  ↓ 完成后归档
04.已归档任务/01.背包系统_v1/

迭代版本：03.正在处理的任务/01.背包系统_v2/
  ↓ 完成后归档
04.已归档任务/01.背包系统_v2/
```

## 🎯 完整示例

### 示例：开发背包系统

#### Step 1: 策划讨论
```
你：我想开发一个背包系统，支持 100 格容量...
我：背包可以升级吗？最多支持多少格？
你：支持，最多 200 格
...讨论 3 轮...
你：我满意了，可以开始了

输出：工作/1.策划/01.背包系统/策划案.md
```

#### Step 2: 需求拆解（自动执行）
```
系统自动：
  ├─ 需求Agent 拆解需求
  │  → 工作/2.需求列表/01.背包系统/需求.md
  │  → 工作/3.正在处理的任务/01.背包系统/（OpenSpec 工作目录）
  └─ 美术Agent 分析美术资源
     → 工作/1.美术/01.背包系统/美术需求分析.md
     → 工作/1.美术/01.背包系统/提示词.md
```

#### Step 3: 执行追踪
```
开发者按照 design.md 实现功能
  ↓ 每完成一个阶段
更新 tasks.md 进度
  ↓ 全部实现完成
记录 实现记录/Phase1.md, Phase2.md...
```

#### Step 4: 自动化测试
```
系统自动：
  ├─ 设计测试方案
  ├─ 编写测试脚本
  └─ 运行测试验证
  
输出：
  → 工作/3.测试/01.背包系统/测试方案.md
  → 工作/3.测试/01.背包系统/自动化测试脚本.cs
  → 工作/3.测试/01.背包系统/测试结果.md
```

#### Step 5: 归档完成
```
生成完成清单
  ↓
移动到 工作/4.已归档任务/01.背包系统_v1/
  ↓
✅ 完成！
```

## ❓ FAQ

### Q: 第一次开发时，我需要做什么？
**A**: 只需要描述需求即可。剩下的拆解、美术分析、自动化测试都由系统处理。

### Q: 如果需求变化了怎么办？
**A**: 在 Phase 1 或 Phase 2 中随时可以修改。一旦进入 Phase 3，大的需求变化应该等本版本完成后创建 v2。

### Q: 测试失败了怎么办？
**A**: 回到 Phase 3，修复代码，然后重新运行测试。

### Q: 如何迭代功能？
**A**: 完成 v1 后，创建新的 Phase 2 工作目录（NN.功能名_v2），重复整个流程。
